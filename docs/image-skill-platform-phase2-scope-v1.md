# 图像 Skill 平台 V1 · 第二阶段开发目标与范围

## 文档说明

- 文档目标：在第一阶段复盘基础上，正式定义第二阶段的开发目标、范围边界、优先级和不做项。
- 文档定位：承接 [image-skill-platform-phase1-review-v1.md](./image-skill-platform-phase1-review-v1.md) 的第二阶段范围文档。
- 当前状态：第二阶段范围主文档。
- 更新时间：2026-05-05。

## 1. 第二阶段结论先行

第二阶段不再解决“工作台有没有双模式”这个问题，而是解决：

- 普通版够不够轻、够不够顺
- 专业版够不够完整、够不够可控
- 模板和轻量追问有没有真正接入主流程

一句话总结：

第一阶段解决“站起来”，第二阶段解决“更像产品”。

## 2. 为什么现在进入第二阶段

第一阶段已经完成了这些基础条件：

- 双模式统一工作台已成立
- 普通版主链路已跑通
- 专业版最小控制视图已成立
- 作品和任务都已能回流到工作台
- 模板页和配置中心已完成第一轮定位调整

因此，当前最合理的下一步不是继续铺更多页面，而是进入体验增强和能力收口阶段。

## 3. 第二阶段总目标

第二阶段总目标分成 3 句：

1. 让普通版更像一个普通用户真的愿意持续使用的图片产品
2. 让专业版更像一个高频用户愿意长期停留的创作控制台
3. 让模板和轻量追问开始真正驱动工作台，而不只是挂在边上

## 4. 第二阶段产品原则

- 普通版继续去专业化，不往回长成旧控制台
- 专业版继续系统化，不停留在“多几个信息块”
- 模板优先接入主流程，不优先做更大的模板后台
- 追问优先做轻量可用，不优先做复杂对话机器人
- 版本链优先做可理解，不优先做复杂图谱可视化
- 第二阶段继续控范围，不扩散到团队协作、项目系统、外部 API

## 5. 第二阶段范围总览

建议把第二阶段收口成 4 个方向。

| 方向 | 名称 | 目标 | 优先级 |
|---|---|---|---|
| S1 | 普通版体验增强 | 继续让普通版更轻、更顺、更像消费产品 | P0 |
| S2 | 专业版控制链增强 | 让专业版更完整、更可追溯、更可复用 | P0 |
| S3 | 模板系统升级 | 让模板更像结构化模板，而不是纯 Prompt 存档 | P1 |
| S4 | 轻量追问与版本表达 | 让普通版少想一步，让版本链更容易理解 | P1 |

## 6. S1 普通版体验增强

### 6.1 目标

把当前普通版从“已经成立”推进到“真正顺手”。

### 6.2 当前问题

- 输入区仍残留旧 Prompt / 参数体系影子
- 普通版视觉层级还不够明确
- 某些提示和状态仍偏工程化
- 结果动作虽然有了，但动作后的衔接还可以更顺

### 6.3 第二阶段内建议做

- 继续压缩普通版输入层复杂度
- 更清晰地区分“任务入口 / 输入区 / 结果区 / 下一步动作”
- 优化普通版生成前后的引导文案
- 优化结果动作后的状态反馈
- 进一步隐藏或翻译残余专业术语

### 6.4 第二阶段内不要求完成

- 不要求普通版一开始就接完整动态追问系统
- 不要求普通版完全替代专业版的所有入口

## 7. S2 专业版控制链增强

### 7.1 目标

把当前“最小控制视图”推进成“有完整控制感的专业版”。

### 7.2 当前问题

- 参数快照不够系统
- Prompt 预览已经有，但编辑、对比、复用链不完整
- Provider / model 可见，但执行解释还不够强
- 缺少“基于当前结果重跑 / 对比 / 复用”的完整控制链

### 7.3 第二阶段内建议做

- 增强参数快照和参数摘要区
- 增强 Prompt 预览与复制链
- 增强 Provider / model / 执行状态说明
- 让专业版从结果页回到参数控制更顺
- 让模板与专业版输入区的关系更明确

### 7.4 第二阶段内不要求完成

- 不要求做复杂对比器
- 不要求做完整高级调试台

## 8. S3 模板系统升级

### 8.1 目标

让模板从“Prompt 模板库”逐步升级成“结构模板入口”。

### 8.2 当前问题

- 当前模板页更像资产页，而不是模板驱动页
- 模板只有标题、内容、分类、标签
- 模板没有真正控制工作台里的追问、参数和入口

### 8.3 第二阶段内建议做

- 强化模板分类结构
- 增加模板说明、适用场景、推荐入口
- 把模板更明显地接到普通版任务入口
- 把模板更明显地接到专业版控制面板
- 为后续模板结构字段预留前端表达

### 8.4 第二阶段内不要求完成

- 不要求完整模板 DSL 后台
- 不要求完整模板版本管理系统

## 9. S4 轻量追问与版本表达

### 9.1 目标

让普通版减少“自己先想一段话”的负担，让结果链更容易被理解和延续。

### 9.2 当前问题

- 普通版用户仍需要自己先组织描述
- 结果动作已有，但背后没有真正“意图 -> 补全 -> 执行”链
- 作品和任务都能回流，但用户还不够容易理解版本关系

### 9.3 第二阶段内建议做

- 为 2 到 3 个高频场景接入轻量追问
- 追问优先用按钮和结构选项，不走重聊天
- 增强“当前结果来自哪里”的表达
- 增强“继续这一版 / 从这一版重试”的表达

### 9.4 第二阶段内不要求完成

- 不要求复杂版本图谱
- 不要求完整动态问答引擎

## 10. 第二阶段最值得优先做的 6 个任务

如果把第二阶段进一步收口，我建议优先就是这 6 项：

1. 普通版输入区继续收口，减少残余专业表达
2. 普通版结果动作链继续做顺，降低下一步犹豫成本
3. 专业版参数快照与 Prompt 链增强
4. 模板页升级为更像模板库的结构视图
5. 为 2 到 3 个高频场景接轻量动态追问
6. 版本链表达增强，让用户更容易看懂来源与延续关系

## 11. 第二阶段明确不做

为了防止第二阶段再次发散，明确不做以下内容：

- 团队协作
- 项目空间
- 复杂权限体系
- 外部 API 平台
- 大规模自动工作流
- 完整模板 CMS 后台
- 企业级审批流

这些能力不是不重要，而是不应该在当前阶段抢主线。

## 12. 第二阶段成功标准

第二阶段完成后，至少应达到：

- 普通版明显更轻，普通用户更不容易被参数打断
- 专业版明显更完整，进阶用户能更顺地复用和重跑
- 模板在前台的角色更强，不只是保存 Prompt
- 至少有少量场景具备轻量追问体验
- 用户更容易理解“这张图是从哪一版继续来的”

## 13. 第二阶段工程策略

第二阶段建议继续沿用第一阶段的工程策略：

- 继续在 `feature/image-skill-platform-v1` 总分支上集成
- 继续拆多 Agent 子任务
- 继续先做局部目标清晰的改造，再做总分支集成验收
- 每一轮都跑 `typecheck`、`test:smoke:client`、`build`

## 14. 第二阶段总结

第二阶段的核心不是做更多，而是做更准。

要把第一阶段已经搭起来的骨架真正变成产品，就必须把普通版、专业版、模板和轻量追问这四条线再往前推一步，但仍然守住边界。

如果继续沿用多 Agent 方式，下一步最自然的动作就是：

- 立第二阶段 backlog
- 再拆第二阶段 Agent 分工
