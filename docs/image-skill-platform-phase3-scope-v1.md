# 图像 Skill 平台 V1 · 第三阶段开发目标与范围

## 文档说明

- 文档目标：在第二阶段复盘基础上，正式定义第三阶段的开发目标、范围边界、优先级和不做项。
- 文档定位：承接 [image-skill-platform-phase2-review-v1.md](./image-skill-platform-phase2-review-v1.md) 的第三阶段范围文档。
- 当前状态：第三阶段范围主文档。
- 更新时间：2026-05-05。

## 1. 第三阶段结论先行

第三阶段不再继续主要解决“前台够不够顺、够不够轻”的问题，而是开始解决：

- 模板能不能真正承载结构化输入
- 轻量追问能不能从提示升级成正式输入流程
- 版本链能不能从来源提示升级成工作流能力
- 专业版能不能进一步承接结构化控制链

一句话总结：

第一阶段解决“站起来”，第二阶段解决“更像产品”，第三阶段解决“底层能力开始真正成立”。

## 2. 为什么现在进入第三阶段

经过前两阶段之后，当前已经具备以下前提：

- 双模式统一工作台已经成立
- 普通版已具备可用的任务入口、输入链和结果动作链
- 专业版已具备最小控制链和第二轮增强后的执行上下文表达
- 模板页已经从 Prompt 存档页走向模板库认知
- 轻量追问已经接近主流程，但还未正式结构化
- 作品与任务回流已经成立，版本来源表达已有基础

因此，当前最合理的推进方向，不是继续只做表层抛光，而是把前台已经暴露出来的产品承诺，逐步补到底层结构层。

## 3. 第三阶段总目标

第三阶段总目标可以收口成 3 句：

1. 让模板开始成为真正的结构化工作台入口，而不只是可复用文案资产
2. 让轻量追问开始成为正式输入机制，而不只是引导性提示
3. 让版本链和专业控制链更紧地接到实际创作和重跑流程里

## 4. 第三阶段产品原则

- 先补结构，不先继续铺更多表层页面
- 先做少量高频场景的结构化闭环，不先追求全场景覆盖
- 追问优先结构化和可落盘，不优先做更像聊天机器人的体验
- 模板优先控制输入和入口，不优先做庞大的模板后台系统
- 版本链优先服务“继续创作”和“再次重跑”，不优先做复杂图谱展示
- 专业版优先补控制链闭环，不优先做更大的专家参数面板
- 第三阶段继续强控范围，不扩散到协作、项目系统、开放平台

## 5. 第三阶段范围总览

建议把第三阶段收口成 5 个方向。

| 方向 | 名称 | 目标 | 优先级 |
|---|---|---|---|
| P31 | 模板结构化 | 让模板具备字段、场景、入口、展示的结构化基础 | P0 |
| P32 | 轻量追问正式化 | 让高频场景具备可保存、可回放的结构化追问输入链 | P0 |
| P33 | 版本链工作流增强 | 让来源提示升级为真正的继续创作、重跑和分叉入口 | P0 |
| P34 | 专业版控制链深化 | 让专业版更接近可复用、可重跑、可对照的创作控制台 | P1 |
| P35 | 模板-追问-执行链路对齐 | 让普通版、专业版、模板、结果动作之间形成统一语义 | P1 |

## 6. P31 模板结构化

### 6.1 目标

让模板从“内容模板”向“结构模板”过渡，至少建立前台和前端状态层可识别的字段体系。

### 6.2 当前问题

- 当前模板仍以 Prompt 内容组织为主
- 模板分类已有，但模板字段结构还不明确
- 模板与工作台的关系主要靠页面入口和说明文案建立
- 模板难以真正驱动追问和执行上下文

### 6.3 第三阶段内建议做

- 定义首批结构模板元信息
- 定义模板字段、场景、入口方式、推荐模式等前端可用结构
- 让模板列表和模板详情开始展示结构信息
- 让工作台能读取首批结构模板基础字段
- 为后续模板版本化和 DSL 化留出前端接口

### 6.4 第三阶段内不要求完成

- 不要求完成完整模板 DSL 编辑器
- 不要求完成模板后台 CMS
- 不要求完成模板版本管理后台

## 7. P32 轻量追问正式化

### 7.1 目标

让轻量追问从“提示式补一句”升级到“基于场景和模板的正式结构化补全流程”。

### 7.2 当前问题

- 当前更多是 follow-up 提示和场景引导
- 缺少可保存的提问步骤定义
- 缺少结构化回答和状态回放
- 缺少追问结果与工作台执行参数的稳定映射

### 7.3 第三阶段内建议做

- 选 2 到 3 个高频场景作为首批正式追问场景
- 为每个场景定义问题顺序、选项和默认值策略
- 让追问结果能写回工作台输入上下文
- 让追问结果能进入历史、重放和继续创作
- 让结果动作能反向触发追问分支

### 7.4 第三阶段内不要求完成

- 不要求做通用问答引擎
- 不要求做自由对话式 Agent 追问
- 不要求所有场景都接入追问

## 8. P33 版本链工作流增强

### 8.1 目标

让版本链从“看得懂来源”升级到“可以沿来源继续创作、回退、重跑和分叉”。

### 8.2 当前问题

- 当前主要还是来源提示和回流提示增强
- 用户能知道从哪里来，但还不够容易基于版本关系继续操作
- 作品、任务、结果动作之间的版本语义仍有断点

### 8.3 第三阶段内建议做

- 强化“继续这一版”“从这一版重试”“从这一版分叉”的操作表达
- 统一作品页、任务页、结果页中的版本语义
- 增强版本来源摘要与父节点信息
- 让普通版和专业版都能更明确地消费版本上下文
- 为后续版本图谱和参数差异对比预留结构

### 8.4 第三阶段内不要求完成

- 不要求复杂图谱可视化
- 不要求完整版本比较器
- 不要求项目级版本编排

## 9. P34 专业版控制链深化

### 9.1 目标

让专业版更接近真正可复用、可重跑、可对照的控制工作台。

### 9.2 当前问题

- Prompt、参数、Provider 已可见，但对照关系还不够强
- 缺少更清晰的结构模板上下文
- 重跑、派生、回退仍以流程感为主，控制感还不够完整

### 9.3 第三阶段内建议做

- 强化结构模板在专业版中的展示
- 强化 Prompt、参数、Provider、来源之间的对应关系
- 强化从当前结果回到重跑和派生的控制入口
- 增强参数快照的可读性与复用提示
- 为后续差异对比留出结构位

### 9.4 第三阶段内不要求完成

- 不要求完成复杂多版本 diff
- 不要求完成专家级参数实验室

## 10. P35 模板-追问-执行链路对齐

### 10.1 目标

避免第三阶段做出几个各自成立但彼此割裂的模块，而是让模板、追问、工作台输入、结果动作和执行上下文说同一种语言。

### 10.2 当前问题

- 模板、追问、结果动作各自已经有认知基础
- 但字段命名、入口路径、状态回写还不够统一
- 普通版与专业版也可能逐步出现结构不一致

### 10.3 第三阶段内建议做

- 定义首批统一场景对象和字段命名
- 对齐普通版、专业版、模板页的入口参数
- 对齐追问结果、Prompt 填充和结果动作的语义映射
- 对齐版本链动作与来源类型

### 10.4 第三阶段内不要求完成

- 不要求形成最终中台 schema
- 不要求一次性统一全站所有历史数据结构

## 11. 第三阶段最值得优先做的 6 个任务

如果把第三阶段进一步收口，我建议优先就是这 6 项：

1. 为首批模板定义结构化元信息和字段基础
2. 为 2 到 3 个高频场景接正式的轻量追问步骤
3. 让追问结果可保存、可回放、可写回工作台
4. 让“继续这一版 / 重试这一版 / 从这一版分叉”形成明确动作体系
5. 让专业版更清楚展示模板、Prompt、参数、Provider、来源之间的关系
6. 让模板页、工作台、结果页说同一套场景与字段语言

## 12. 第三阶段明确不做

为了防止第三阶段再次失控，明确不做以下内容：

- 团队协作
- 项目空间
- 复杂权限体系
- 外部 API 平台
- 自动工作流编排平台
- 模板后台 CMS
- 模板版本管理后台
- 企业级审批流

这些能力后续可以做，但不是当前阶段最有价值的结构性补强。

## 13. 第三阶段成功标准

第三阶段完成后，至少应达到：

- 首批模板已经具备结构化字段表达
- 至少 2 到 3 个高频场景具备正式轻量追问流程
- 追问结果能真正回写工作台和历史上下文
- 版本链不只解释来源，还能承接继续创作和分叉动作
- 专业版更清楚地承接结构模板、参数和执行链
- 普通版、专业版、模板页和结果动作之间的场景语义更一致

## 14. 第三阶段工程策略

第三阶段建议继续沿用前两阶段的工程策略：

- 继续在 `feature/image-skill-platform-v1` 总分支上集成
- 继续拆多 Agent 子任务
- 继续先做单条能力链清晰的局部改造，再做总分支集成验收
- 每一轮都跑 `typecheck`、`test:smoke:client`、`build`

但第三阶段要额外强调一点：

- 所有前台体验改动，尽量同步补最小必要的数据结构和状态结构

否则前台体验会继续领先底层结构，后面会出现更重的返工。

## 15. 第三阶段总结

第三阶段的重点，不再是“再顺一点”，而是“开始真正有底层能力”。

这意味着我们要开始把过去两阶段已经讲给用户听的产品故事，逐步补成真正成立的结构：

- 模板不是只有文案
- 追问不是只有提示
- 版本不是只有来源
- 专业版不是只有更多信息

如果继续沿用多 Agent 方式，下一步最自然的动作就是：

- 立第三阶段 backlog
- 再拆第三阶段 Agent 分工
