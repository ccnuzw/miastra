# 图像 Skill 平台 V1 · 第四阶段开发目标与范围

## 文档说明

- 文档目标：在第三阶段复盘基础上，正式定义第四阶段的开发目标、范围边界、优先级和不做项。
- 文档定位：承接 [image-skill-platform-phase3-review-v1.md](./image-skill-platform-phase3-review-v1.md) 的第四阶段范围文档。
- 当前状态：第四阶段范围主文档。
- 更新时间：2026-05-05。

## 1. 第四阶段结论先行

第四阶段不再主要解决“结构有没有开始成立”的问题，而是开始解决：

- 模板、追问、执行、版本之间有没有统一的共享 schema
- 轻量追问能不能从首批场景能力升级成可扩展机制
- 版本链能不能从动作语义升级成更稳定的多轮生产工作流
- 专业版能不能真正服务高频复用和结构化重跑

一句话总结：

第一阶段解决“站起来”，第二阶段解决“更像产品”，第三阶段解决“结构开始成立”，第四阶段解决“结构真正打通并可扩展”。

## 2. 为什么现在进入第四阶段

经过前三阶段之后，当前已经具备以下前提：

- 双模式统一工作台已经稳定成立
- 模板已经拥有首批结构字段和元信息
- 轻量追问已经具备快照和状态回写基础
- 版本链已经具备继续、重试、分叉的动作语义
- 专业版已经具备结构模板、Prompt、参数、Provider、来源的基础解释链

因此，当前最合理的推进方向，不是再做更多局部结构点，而是把这些结构点打成一条更统一的能力链。

## 3. 第四阶段总目标

第四阶段总目标可以收口成 3 句：

1. 让模板、追问、执行、版本开始共享同一套场景与字段对象
2. 让轻量追问从少量手工场景走向可扩展的配置机制
3. 让专业版和版本链更好地服务高频复用、重跑和派生

## 4. 第四阶段产品原则

- 优先统一共享 schema，不优先继续分散补点
- 优先配置化和扩展性，不优先继续堆更多硬编码场景
- 优先增强多轮工作流，不优先增加新的静态页面
- 优先让专业版服务高频复用，不优先继续堆展示信息
- 继续守住普通版和专业版边界，不让结构能力反向压重普通版
- 第四阶段继续强控范围，不扩散到协作、项目系统、开放平台

## 5. 第四阶段范围总览

建议把第四阶段收口成 5 个方向。

| 方向 | 名称 | 目标 | 优先级 |
|---|---|---|---|
| P41 | 统一场景 Schema 深化 | 让模板、追问、执行、版本共享更统一的对象和字段命名 | P0 |
| P42 | 轻量追问配置化 | 让追问场景、步骤和默认值更容易扩展 | P0 |
| P43 | 版本链生产流增强 | 让多轮继续、重试、分叉具备更稳定的工作流心智 | P0 |
| P44 | 专业版高频复用增强 | 让专业版更明确服务结构复用、重跑和派生 | P1 |
| P45 | 模板向 Skill 入口演进 | 让模板从结构入口进一步走向带执行意图的 Skill 入口 | P1 |

## 6. P41 统一场景 Schema 深化

### 6.1 目标

让模板结构、追问结果、执行参数、版本上下文不再主要依赖多层适配，而是共享更稳定的场景对象和字段命名。

### 6.2 当前问题

- 结构已经起步，但还不是完整统一 schema
- 多处仍有适配层和命名转换
- 普通版、专业版、模板页之间仍有部分字段表达差异

### 6.3 第四阶段内建议做

- 定义统一的场景对象与字段命名约束
- 对齐模板 schema、追问快照、执行参数、版本上下文
- 清理明显重复或分裂的字段表达
- 为后续服务端或持久化扩展预留稳定接口

### 6.4 第四阶段内不要求完成

- 不要求一次性重构全量历史数据
- 不要求形成最终后端中台 schema

## 7. P42 轻量追问配置化

### 7.1 目标

让当前已经成立的追问快照和步骤，不再只服务首批手工场景，而是更容易扩展到新的高频场景。

### 7.2 当前问题

- 场景覆盖还很有限
- 追问步骤和字段策略仍偏手工定义
- 模板字段到追问字段的映射还可进一步稳定

### 7.3 第四阶段内建议做

- 把追问定义抽成更清晰的配置结构
- 让模板字段可以驱动追问步骤和默认值
- 扩展到更多高频场景，但仍控制范围
- 强化追问结果对 Prompt 和结果动作的影响链

### 7.4 第四阶段内不要求完成

- 不要求覆盖所有模板场景
- 不要求做复杂自由对话追问

## 8. P43 版本链生产流增强

### 8.1 目标

让版本链从“动作语义成立”进一步升级为“高频多轮创作也能稳定理解和复用”的工作流。

### 8.2 当前问题

- 继续、重试、分叉的语义已成立，但父子关系表达仍偏轻
- 多轮分叉后的来源心智还不够稳
- 版本链与结构字段的关系还可继续增强

### 8.3 第四阶段内建议做

- 增强父节点、祖先节点和当前节点的摘要表达
- 强化多轮回流、派生和重跑的来源说明
- 强化版本链与模板、追问、参数之间的对应关系
- 让普通版和专业版都能更稳定消费多轮版本上下文

### 8.4 第四阶段内不要求完成

- 不要求做完整版本图谱可视化
- 不要求做复杂差异对比器

## 9. P44 专业版高频复用增强

### 9.1 目标

让专业版更明确服务于“同一场景反复复用、反复重跑、反复派生”的高频创作模式。

### 9.2 当前问题

- 专业版现在更能解释当前执行，但复用感还不够强
- 模板结构字段与最终执行的对应关系还可继续增强
- 参数快照和派生入口还可以更面向高频工作流

### 9.3 第四阶段内建议做

- 强化结构字段与最终 Prompt / 执行的对应关系
- 强化参数快照与版本派生的联动
- 强化专业版中对模板入口、重跑入口、派生入口的复用表达
- 让专业版更像“结构化生产控制台”

### 9.4 第四阶段内不要求完成

- 不要求做专家级实验台
- 不要求做复杂多版本比对面板

## 10. P45 模板向 Skill 入口演进

### 10.1 目标

让模板不只是结构入口，还开始具备更明确的执行意图、推荐模式和工作台进入策略。

### 10.2 当前问题

- 模板已经结构化，但仍主要表达内容和字段
- 模板与执行策略、入口策略的关系还不够强
- 还没有真正走到“Skill 入口”的层面

### 10.3 第四阶段内建议做

- 强化模板的推荐模式和推荐入口表达
- 强化模板到普通版/专业版/结果动作的进入策略
- 强化模板与版本链、追问链的关联表达
- 让模板更接近可执行 Skill 入口，而不是静态资源页

### 10.4 第四阶段内不要求完成

- 不要求做完整 Skill 市场
- 不要求做模板发布后台

## 11. 第四阶段最值得优先做的 6 个任务

1. 统一模板、追问、执行、版本的共享场景 schema
2. 让追问配置从手工定义升级成更稳定的配置机制
3. 扩展并增强追问结果到 Prompt 和结果动作的闭环
4. 让版本链具备更稳定的多轮父子摘要和分叉心智
5. 让专业版更明确服务结构复用、重跑和派生
6. 让模板更接近 Skill 入口，而不是只停留在结构入口

## 12. 第四阶段明确不做

为了防止第四阶段再次发散，明确不做以下内容：

- 团队协作
- 项目空间
- 复杂权限体系
- 外部 API 平台
- 自动工作流编排平台
- 模板后台 CMS
- 企业级审批流
- 社区化模板市场

这些能力后续可以做，但不应该抢当前结构统一和机制扩展的主线。

## 13. 第四阶段成功标准

第四阶段完成后，至少应达到：

- 模板、追问、执行、版本之间拥有更稳定的统一场景 schema
- 轻量追问更容易扩展到新的高频场景
- 多轮版本链更容易理解和继续创作
- 专业版更明显服务高频复用、重跑和派生
- 模板更明显具备 Skill 入口属性

## 14. 第四阶段工程策略

第四阶段建议继续沿用前三阶段的工程策略：

- 继续在 `feature/image-skill-platform-v1` 总分支上集成
- 继续拆多 Agent 子任务
- 继续每轮都跑 `typecheck`、`test:smoke:client`、`build`

但第四阶段要额外强调两点：

- 优先减少命名和结构分裂，而不是继续堆新对象
- 新增结构时尽量同步补最小必要的测试

## 15. 第四阶段总结

第四阶段的重点，是把第三阶段已经成立的“结构点”，继续打通成“结构链”。

这意味着下一步最有价值的，不是再开更多页面，而是持续推进：

- schema 更统一
- 追问更可扩展
- 版本流更稳定
- 专业版更可复用
- 模板更像 Skill 入口
