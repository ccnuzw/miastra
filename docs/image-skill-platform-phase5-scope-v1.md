# 图像 Skill 平台 V1 · 第五阶段开发目标与范围

## 文档说明

- 文档目标：在第四阶段复盘基础上，正式定义第五阶段的开发目标、范围边界、优先级和不做项。
- 文档定位：承接 [image-skill-platform-phase4-review-v1.md](./image-skill-platform-phase4-review-v1.md) 的第五阶段范围文档。
- 当前状态：第五阶段范围主文档。
- 更新时间：2026-05-05。

## 1. 第五阶段结论先行

第五阶段不再主要解决“共享 schema 有没有被压得更统一”的问题，而是开始解决：

- 共享 schema 能不能进一步变成更稳定的系统级契约
- 轻量追问能不能从配置化升级到更可组合的机制
- 版本链能不能从可用的生产流升级到高频场景下更高效的生产流
- 专业版能不能从复用工作台升级到更适合对照和校准的工作台
- 模板能不能更明确承接 Skill 运行意图，而不只是入口说明

一句话总结：

第三阶段让结构开始成立，第四阶段让结构更统一，第五阶段开始让这些统一结构更像长期稳定的系统能力。

## 2. 为什么现在进入第五阶段

经过前四阶段之后，当前已经具备以下前提：

- 双模式统一工作台已经稳定可用
- 模板、追问、执行、版本之间已经具备更稳定的共享场景语义
- 轻量追问已经具备配置化雏形
- 版本链已经开始接近多轮生产工作流
- 专业版已经开始服务高频复用和派生
- 模板页已经开始更像 Skill 入口集合

因此，当前最合理的推进方向，不是继续只做前台补点，而是让这些能力更系统化、更契约化、更适合长期演进。

## 3. 第五阶段总目标

第五阶段总目标可以收口成 3 句：

1. 让模板、追问、执行、版本的共享对象更接近系统级契约
2. 让轻量追问和版本链更适合高频创作与扩展
3. 让专业版和模板页更明确承接对照、校准和 Skill 运行心智

## 4. 第五阶段产品原则

- 优先稳定共享契约，不优先继续增加更多散点能力
- 优先可组合和可扩展，不优先继续硬编码扩场景
- 优先高频场景效率，不优先增加新的静态说明层
- 优先让专业版更适合对照和校准，不优先继续增加只读信息
- 优先让模板更接近运行入口，不优先继续堆入口说明
- 第五阶段继续强控范围，不扩散到协作、项目系统、开放平台

## 5. 第五阶段范围总览

建议把第五阶段收口成 5 个方向。

| 方向 | 名称 | 目标 | 优先级 |
|---|---|---|---|
| P51 | 共享契约深化 | 让模板、追问、执行、版本更接近稳定系统契约 | P0 |
| P52 | 追问机制可组合化 | 让追问更容易由字段、场景和默认策略自动组合 | P0 |
| P53 | 版本流高频效率增强 | 让多轮继续、重试、分叉在高频场景下更快更稳 | P0 |
| P54 | 专业版对照与校准增强 | 让专业版更适合比较、校准和派生 | P1 |
| P55 | 模板 Skill 运行意图增强 | 让模板更明确决定追问路径、进入策略和结果动作优先级 | P1 |

## 6. P51 共享契约深化

### 6.1 目标

让模板、追问、执行、版本之间的共享对象，从前端域模型进一步走向更稳定的系统级契约。

### 6.2 当前问题

- 共享 schema 已经更统一，但仍主要成立在前端代码层
- 仍存在部分适配层和桥接逻辑
- 不同层之间的结构边界还可以更明确

### 6.3 第五阶段内建议做

- 收口共享对象边界和字段约束
- 减少重复映射和桥接逻辑
- 强化结构对象在主流程和历史回流中的一致性
- 为后续持久化和服务端承接预留更稳的接口

### 6.4 第五阶段内不要求完成

- 不要求做后端重构
- 不要求一次性统一全部历史数据

## 7. P52 追问机制可组合化

### 7.1 目标

让轻量追问不只是“更容易扩展”，而是更容易由模板字段、场景和默认策略自动组合。

### 7.2 当前问题

- 追问已经配置化，但仍有较多手工定义
- 模板字段与问题生成仍不够自动
- 默认值和推荐值策略仍偏静态

### 7.3 第五阶段内建议做

- 强化字段驱动问题生成
- 抽象默认值和推荐值策略
- 继续增强追问结果到 Prompt、结果动作、回流状态的映射
- 在控制范围内扩更多高频场景

### 7.4 第五阶段内不要求完成

- 不要求做复杂自由对话
- 不要求做全场景自动追问系统

## 8. P53 版本流高频效率增强

### 8.1 目标

让版本链在高频多轮创作场景中，不只是能理解，而且能更快消费和更快操作。

### 8.2 当前问题

- 当前仍然有较多文字摘要依赖
- 高级用户多轮创作时，仍缺更直接的节点心智
- 版本节点与结构字段的关系还可以更显式

### 8.3 第五阶段内建议做

- 强化更结构化的节点摘要
- 强化高频多轮回流下的来源判断
- 强化模板、追问、参数和版本节点之间的显式对应关系
- 优化普通版和专业版对多轮版本流的消费效率

### 8.4 第五阶段内不要求完成

- 不要求做复杂图谱
- 不要求做重量级多版本管理器

## 9. P54 专业版对照与校准增强

### 9.1 目标

让专业版不只是更适合复用，而是更适合比较、校准和派生。

### 9.2 当前问题

- 结构字段与最终 Prompt 的对应关系仍可以更直观
- 当前版与来源版的关系虽能复用，但还不够便于比较
- 参数快照虽更强，但差异感知仍弱

### 9.3 第五阶段内建议做

- 强化结构字段与 Prompt 的对照表达
- 强化当前版与来源版的对照表达
- 强化参数快照间的校准和提示能力
- 强化专业版中的派生与对照并存体验

### 9.4 第五阶段内不要求完成

- 不要求做完整 diff 工作台
- 不要求做复杂实验台

## 10. P55 模板 Skill 运行意图增强

### 10.1 目标

让模板不仅是 Skill 入口集合，还更明确决定追问路径、进入策略和结果动作优先级。

### 10.2 当前问题

- 模板已经更像入口，但运行心智仍偏说明性
- 追问、执行策略和结果动作优先级还没有被模板更强地主导
- 模板对工作台行为的控制还可以继续增强

### 10.3 第五阶段内建议做

- 强化模板对追问路径的影响
- 强化模板对普通版 / 专业版进入策略的影响
- 强化模板对结果动作优先级和默认分支的影响
- 让模板更像实际运行中的 Skill 单元

### 10.4 第五阶段内不要求完成

- 不要求做完整 Skill 市场
- 不要求做模板发布运营系统

## 11. 第五阶段最值得优先做的 6 个任务

1. 继续收口共享对象边界，减少桥接和重复映射
2. 让字段更直接驱动追问问题和默认策略
3. 强化追问结果到执行链和结果动作的自动映射
4. 让多轮版本链更具结构化节点心智
5. 让专业版更适合对照来源、校准参数和派生结果
6. 让模板更明确决定追问路径、进入方式和结果动作优先级

## 12. 第五阶段明确不做

为了防止第五阶段再次发散，明确不做以下内容：

- 团队协作
- 项目空间
- 复杂权限体系
- 外部 API 平台
- 自动工作流编排平台
- 模板后台 CMS
- Skill 市场运营系统
- 企业级审批流

这些能力后续可以做，但不应抢当前系统能力稳定化的主线。

## 13. 第五阶段成功标准

第五阶段完成后，至少应达到：

- 共享对象更接近稳定系统契约
- 追问机制更接近字段驱动和可组合扩展
- 多轮版本流在高频场景下更高效
- 专业版更适合对照、校准和派生
- 模板更明确承接 Skill 运行意图

## 14. 第五阶段工程策略

第五阶段建议继续沿用前四阶段的工程策略：

- 继续在 `feature/image-skill-platform-v1` 总分支上集成
- 继续拆多 Agent 子任务
- 继续每轮都跑 `typecheck`、`test:smoke:client`、`build`

但第五阶段要额外强调两点：

- 优先减少桥接和语义分裂，不优先继续堆新层
- 新增机制优先配测试，不要只靠手工验证

## 15. 第五阶段总结

第五阶段的重点，是把第四阶段已经较统一的结构能力，继续推进成更稳定、更契约化、更高频可用的系统能力。

这意味着下一步最有价值的，不是继续开更多说明页，而是持续推进：

- 共享契约更稳
- 追问更可组合
- 版本流更高效
- 专业版更可对照
- 模板更像真正的 Skill 单元
