# 图像 Skill 平台 V1 · 第六阶段开发目标与范围

## 文档说明

- 文档目标：在第五阶段复盘基础上，正式定义第六阶段的开发目标、范围边界、优先级和不做项。
- 文档定位：承接 [image-skill-platform-phase5-review-v1.md](./image-skill-platform-phase5-review-v1.md) 的第六阶段范围文档。
- 当前状态：第六阶段范围主文档。
- 更新时间：2026-05-05。

## 1. 第六阶段结论先行

第六阶段不再主要解决“运行时契约有没有出现”的问题，而是开始解决：

- generation contract 能不能继续从前端运行时对象走向更完整的全链路契约
- 模板 runtime 能不能更深地决定追问路径、进入策略和结果动作分支
- 版本流能不能从结构化摘要走向更高效的差异消费
- 专业版能不能从对照工作台走向更强的校准决策工作台

一句话总结：

第五阶段让系统开始 contract 化和 runtime 化，第六阶段开始让这些 contract 和 runtime 更深地接管主流程。

## 2. 为什么现在进入第六阶段

经过前五阶段之后，当前已经具备以下前提：

- 模板、追问、执行、版本之间已有更稳定的共享场景语义
- generation contract 已经出现，并开始承接主流程
- 模板 runtime 已经开始驱动默认追问和动作优先级
- 版本链已经能基于 contract 恢复更多运行上下文
- 专业版已经开始具备对照和校准能力

因此，当前最合理的推进方向，不是继续只做配置扩展，而是让 contract、runtime、版本流和专业版校准能力更深地介入真实创作链。

## 3. 第六阶段总目标

第六阶段总目标可以收口成 3 句：

1. 让 contract 更像全链路主干，而不只是前端运行时中间层
2. 让模板 runtime 更像真正的运行决策器，而不只是默认值提供器
3. 让版本流与专业版更适合高频生产下的差异判断和下一步决策

## 4. 第六阶段产品原则

- 优先加深主流程承接，不优先继续扩展外围说明层
- 优先 runtime 决策力，不优先继续堆静态配置
- 优先差异消费效率，不优先继续补更多文字摘要
- 优先高频专业场景可判断性，不优先继续增加只读上下文
- 继续守住普通版与专业版边界，不让复杂决策流回灌普通版
- 第六阶段继续强控范围，不扩散到协作、项目系统、开放平台

## 5. 第六阶段范围总览

建议把第六阶段收口成 5 个方向。

| 方向 | 名称 | 目标 | 优先级 |
|---|---|---|---|
| P61 | Contract 全链路深化 | 让 generation contract 更像主干契约 | P0 |
| P62 | Runtime 决策深化 | 让模板 runtime 更明确驱动追问、入口和结果分支 | P0 |
| P63 | 版本差异消费增强 | 让高频多轮版本流更容易看出关键变化 | P0 |
| P64 | 专业版校准决策增强 | 让专业版更适合判断下一步该从哪一版派生和如何校准 | P1 |
| P65 | Skill 运行闭环增强 | 让模板、追问、执行、结果动作更像一个闭环运行单元 | P1 |

## 6. P61 Contract 全链路深化

### 6.1 目标

让 generation contract 从前端运行时契约，继续走向更稳定的主干契约。

### 6.2 当前问题

- 当前 contract 仍偏前端运行时层
- 旧 snapshot 与新 contract 之间仍有兼容桥接
- 历史记录、回流链、执行链对 contract 的消费深度还可继续增强

### 6.3 第六阶段内建议做

- 继续减少旧结构兼容分叉
- 强化主流程和回流链对 contract 的统一消费
- 强化 contract 对 scene、prompt、parameters、guidedFlow、references、draw 的主干地位
- 为后续持久化和任务记录承接继续压结构

### 6.4 第六阶段内不要求完成

- 不要求做服务端重构
- 不要求全量历史数据迁移

## 7. P62 Runtime 决策深化

### 7.1 目标

让模板 runtime 更明确决定追问路径、进入方式和结果动作优先级，而不只是提供模板默认值。

### 7.2 当前问题

- runtime 已成立，但决策仍偏轻
- 模板对执行策略和后续动作的主导仍有限
- runtime 对不同来源和模式的控制还不够深

### 7.3 第六阶段内建议做

- 强化模板对追问路径的主导
- 强化模板对普通版 / 专业版进入方式的控制
- 强化模板对结果动作优先级和默认分支的控制
- 强化 runtime 与 contract、版本来源之间的联动

### 7.4 第六阶段内不要求完成

- 不要求做完整策略引擎
- 不要求做复杂模型路由系统

## 8. P63 版本差异消费增强

### 8.1 目标

让版本流不只解释来源，而是更容易让高频用户看出这版和上一版到底差在哪里。

### 8.2 当前问题

- 当前仍然偏依赖结构化摘要
- 差异判断仍不够直观
- 多轮派生时“变化点”消费效率还不够高

### 8.3 第六阶段内建议做

- 强化更直观的结构变化摘要
- 强化当前节点与父节点的差异消费
- 强化模板、追问、参数变化在版本流中的显式表达
- 优化普通版和专业版对差异信息的消费方式

### 8.4 第六阶段内不要求完成

- 不要求做复杂图谱
- 不要求做完整 diff 工具

## 9. P64 专业版校准决策增强

### 9.1 目标

让专业版不仅能对照，还能更快帮助用户决定下一步该怎么改、该从哪一版派生。

### 9.2 当前问题

- 对照能力已有，但决策提示还偏弱
- 结构字段、Prompt、参数和来源版之间还缺更强的“下一步建议”
- 派生和校准虽然能做，但还不够像决策台

### 9.3 第六阶段内建议做

- 强化 field-to-prompt 对照
- 强化 current-vs-source 对照
- 强化参数校准提示
- 强化派生与重跑的决策辅助表达

### 9.4 第六阶段内不要求完成

- 不要求做完整实验室
- 不要求做复杂对照矩阵

## 10. P65 Skill 运行闭环增强

### 10.1 目标

让模板、追问、执行、结果动作和版本回流更像一个完整 Skill 运行闭环，而不是若干相关模块。

### 10.2 当前问题

- 模板 runtime 已有，但闭环感还不够强
- 追问、执行、动作和回流仍存在阶段性断点
- Skill 心智已经出现，但还不够完整

### 10.3 第六阶段内建议做

- 强化模板到追问到执行到结果动作的闭环说明与状态承接
- 强化结果动作对 runtime 分支的反向影响
- 强化版本回流重新进入 Skill 闭环的体验
- 让模板更像运行中的 Skill 单元，而不只是入口

### 10.4 第六阶段内不要求完成

- 不要求做 Skill 市场
- 不要求做运营发布后台

## 11. 第六阶段最值得优先做的 6 个任务

1. 继续减少 contract 与旧结构之间的桥接分裂
2. 让模板 runtime 更明确决定追问和动作分支
3. 强化追问结果到 contract 和执行参数的承接
4. 强化版本节点的结构化差异摘要
5. 强化专业版对照链里的校准和决策支持
6. 让模板更明显成为“运行中的 Skill 单元”

## 12. 第六阶段明确不做

为了防止第六阶段再次发散，明确不做以下内容：

- 团队协作
- 项目空间
- 复杂权限体系
- 外部 API 平台
- 自动工作流编排平台
- 模板后台 CMS
- Skill 市场运营系统
- 企业级审批流

这些能力后续可以做，但不应抢当前 contract 深化和 runtime 决策深化的主线。

## 13. 第六阶段成功标准

第六阶段完成后，至少应达到：

- generation contract 更像统一主干契约
- 模板 runtime 更明确驱动追问、入口和结果动作
- 版本流更容易看出关键变化
- 专业版更适合做校准和下一步决策
- 模板、追问、执行、动作、回流更像完整 Skill 闭环

## 14. 第六阶段工程策略

第六阶段建议继续沿用前五阶段的工程策略：

- 继续在 `feature/image-skill-platform-v1` 总分支上集成
- 继续拆多 Agent 子任务
- 继续每轮都跑 `typecheck`、`test:smoke:client`、`build`

但第六阶段要额外强调两点：

- 优先让 contract 和 runtime 真正承接主流程，不优先增加外围说明
- 新增差异消费和校准能力时优先补测试，避免只靠人工判断

## 15. 第六阶段总结

第六阶段的重点，是把第五阶段已经开始出现的 contract 化、runtime 化、对照化能力，继续推进成真正深度接管主流程的系统能力。

这意味着下一步最有价值的，不是继续堆更多功能面，而是持续推进：

- contract 更主干
- runtime 更有决策力
- 版本流更会解释差异
- 专业版更会帮助决策
- Skill 闭环更完整
