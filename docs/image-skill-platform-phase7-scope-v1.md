# 图像 Skill 平台 V1 · 第七阶段开发目标与范围

## 文档说明

- 文档目标：在第六阶段复盘基础上，正式定义第七阶段的开发目标、范围边界、优先级和不做项。
- 文档定位：承接 [image-skill-platform-phase6-review-v1.md](./image-skill-platform-phase6-review-v1.md) 的第七阶段范围文档。
- 当前状态：第七阶段范围主文档。
- 更新时间：2026-05-05。

## 1. 第七阶段结论先行

第七阶段不再主要解决“contract、runtime、版本流、专业版校准能力有没有建立”的问题，而是开始解决：

- contract 能不能从更像主干，走向更稳定的 V1 统一事实来源
- runtime 能不能从更强默认行为，走向更稳定的主流程分支控制
- 版本流能不能从更容易理解，走向更高频、更低负担的生产消费
- 专业版能不能从更像校准台，走向更像可连续决策的工作流面
- 整个双模式工作台能不能进入 V1 上线前收口状态

一句话总结：

第六阶段让结构更深地接管运行，第七阶段开始让这些结构更稳定、更顺手，并更接近 V1 上线收口。

## 2. 为什么现在进入第七阶段

经过前六阶段之后，当前已经具备以下前提：

- generation contract 已经更像统一主干
- 模板 runtime 已经更明显主导追问、入口和结果动作
- 版本流已经能更结构化地消费来源和变化点
- 专业版已经开始像校准决策台
- 模板、追问、执行、动作、回流之间已经形成更强的 Skill 闭环感

因此，当前最合理的推进方向，不是继续大幅扩新概念，而是把这些结构压得更稳、更快、更适合 V1 上线前使用。

## 3. 第七阶段总目标

第七阶段总目标可以收口成 3 句：

1. 让 contract 和 runtime 更稳定地成为主流程唯一主干
2. 让版本流和专业版更适合高频使用下的快速判断与连续决策
3. 让普通版、专业版和模板入口一起进入 V1 收口阶段

## 4. 第七阶段产品原则

- 优先稳定主干，不优先继续发散外围概念
- 优先高频使用效率，不优先补更多低频说明
- 优先异常态和边界态更稳，不优先新入口扩张
- 优先普通版更自然、专业版更清晰，不优先继续增加展示信息
- 优先 V1 可上线收口，不优先平台远期能力扩张
- 第七阶段继续强控范围，不扩散到团队协作、开放平台、企业功能

## 5. 第七阶段范围总览

建议把第七阶段收口成 5 个方向。

| 方向 | 名称 | 目标 | 优先级 |
|---|---|---|---|
| P71 | Contract 稳定收口 | 让 generation contract 更接近 V1 统一事实来源 | P0 |
| P72 | Runtime 分支收口 | 让模板 runtime 对主流程分支控制更稳定、更少例外 | P0 |
| P73 | 版本高频效率收口 | 让高频多轮版本流更快判断、更快继续 | P0 |
| P74 | 专业版连续决策收口 | 让专业版更适合连续校准、派生和重跑 | P1 |
| P75 | 双模式体验与异常态收口 | 让普通版、专业版、模板入口进入 V1 上线前收口状态 | P1 |

## 6. P71 Contract 稳定收口

### 6.1 目标

让 generation contract 从“更像主干”继续走向“更稳定的 V1 统一事实来源”。

### 6.2 当前问题

- 仍存在少量旧结构兼容桥接
- contract 在部分回流和历史链里仍有适配层痕迹
- 主流程统一性虽然增强，但还没到“默认只认 contract”

### 6.3 第七阶段内建议做

- 继续压缩旧结构兼容分叉
- 强化任务、结果、回流、重跑对 contract 的统一消费
- 明确 contract 在关键主流程中的事实来源地位
- 继续减少 snapshot / fallback / bridge 的重复语义

### 6.4 第七阶段内不要求完成

- 不要求做后端持久化重构
- 不要求做历史全量迁移

## 7. P72 Runtime 分支收口

### 7.1 目标

让模板 runtime 对追问路径、入口方式、结果动作、回流再进入的控制更稳定、更少例外分支。

### 7.2 当前问题

- runtime 决策已经明显增强，但还存在例外分支
- 某些入口仍有手工覆盖或旁路逻辑
- runtime 对结果动作和回流闭环的主导还可继续压实

### 7.3 第七阶段内建议做

- 继续减少 runtime 旁路逻辑
- 强化模板 runtime 对入口、追问、结果动作的统一控制
- 强化 runtime 与 contract、版本来源、场景入口之间的默认联动
- 优化 runtime 决策在普通版和模板入口中的一致性

### 7.4 第七阶段内不要求完成

- 不要求做完整策略引擎平台
- 不要求做复杂模型决策树

## 8. P73 版本高频效率收口

### 8.1 目标

让版本流从“更容易看懂变化”继续走向“更适合高频继续、重试和分叉”。

### 8.2 当前问题

- 差异摘要已经更好，但复杂链路仍偏说明型
- 高频用户在继续、重试、分叉之间的判断成本仍可继续下降
- 版本链与当前动作之间的关系还可以更直接

### 8.3 第七阶段内建议做

- 优化版本差异的高频消费密度
- 强化继续、重试、分叉的动作判断提示
- 强化版本节点与模板、追问、参数变化之间的直达关系
- 让普通版和专业版对版本消费更分层但更顺手

### 8.4 第七阶段内不要求完成

- 不要求做复杂版本图谱
- 不要求做完整 diff 面板系统

## 9. P74 专业版连续决策收口

### 9.1 目标

让专业版从“更会对照和校准”，继续走向“更适合连续决策和高频派生”。

### 9.2 当前问题

- 专业版已经像校准台，但连续决策流仍偏碎
- 派生、重跑、参数校准和 prompt 对照之间还可进一步压成一条链
- 来源版、当前版、目标版之间的心智还可以更短

### 9.3 第七阶段内建议做

- 强化连续派生与重跑的决策提示
- 强化来源版 / 当前版 / 目标版之间的对照关系
- 优化参数、Prompt、执行信息在连续校准中的消费顺序
- 让专业版更像高频生产工具，而不是多块高级信息面板

### 9.4 第七阶段内不要求完成

- 不要求做完整实验台
- 不要求做复杂对照矩阵

## 10. P75 双模式体验与异常态收口

### 10.1 目标

让普通版、专业版、模板入口和回流链在异常态、空态和边界态下更稳，更接近 V1 上线前收口状态。

### 10.2 当前问题

- 主流程已经成立，但异常态和边界态还可继续统一
- 双模式切换、模板进入、结果动作再进入的少数边界还可继续磨平
- 普通版和专业版虽然区分明显，但体验统一性还能更稳

### 10.3 第七阶段内建议做

- 补齐主流程异常态、空态和恢复态
- 统一普通版 / 专业版 / 模板入口的边界反馈
- 优化关键主流程的状态提示、文案和降级路径
- 为 V1 上线前回归和手工验收准备更稳状态

### 10.4 第七阶段内不要求完成

- 不要求做完整发布系统
- 不要求扩展新的大型功能模块

## 11. 第七阶段最值得优先做的 6 个任务

1. 继续减少 contract 与旧结构之间的桥接层
2. 继续减少 runtime 决策中的例外分支
3. 优化版本流在继续、重试、分叉下的高频消费效率
4. 强化专业版连续校准和连续派生的决策链
5. 统一普通版、专业版和模板入口的异常态与边界反馈
6. 为 V1 上线前收口建立更稳的状态与回归基础

## 12. 第七阶段明确不做

为了防止第七阶段继续发散，明确不做以下内容：

- 团队协作
- 项目空间
- 复杂权限体系
- 外部 API 平台
- Skill 市场运营系统
- 模板后台 CMS
- 企业级审批流
- 全新大型功能模块扩张

这些能力可以在 V1 之后再评估，但不应抢当前 V1 收口主线。

## 13. 第七阶段成功标准

第七阶段完成后，至少应达到：

- generation contract 更接近 V1 统一事实来源
- 模板 runtime 对主流程分支控制更稳定
- 版本流更适合高频继续、重试和分叉
- 专业版更适合连续决策和连续校准
- 普通版、专业版、模板入口和回流链更接近 V1 上线前收口状态

## 14. 第七阶段工程策略

第七阶段建议继续沿用前六阶段的工程策略：

- 继续使用总分支 + 多 Agent 并行推进
- 继续以 `typecheck`、`smoke`、`build` 作为每轮集成门禁
- 继续每轮都做复盘，不让范围自然膨胀
- 但第七阶段开始更强调“少做新层，多做收口”

## 15. 第七阶段总结

第七阶段不是再打开更大的平台化想象，而是让前六阶段逐步形成的 contract、runtime、版本流和专业版能力，真正进入 V1 上线前的收口阶段。

一句话收口：

第六阶段解决“结构能不能更深地接管运行”，第七阶段解决“这些结构能不能足够稳、足够顺、足够适合上线前收口”。
