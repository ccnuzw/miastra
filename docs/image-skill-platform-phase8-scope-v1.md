# 图像 Skill 平台 V1 · 第八阶段开发目标与范围

## 文档说明

- 文档目标：在第七阶段复盘基础上，正式定义第八阶段的开发目标、范围边界、优先级和不做项。
- 文档定位：承接 [image-skill-platform-phase7-review-v1.md](./image-skill-platform-phase7-review-v1.md) 的第八阶段范围文档。
- 当前状态：第八阶段范围主文档。
- 更新时间：2026-05-05。

## 1. 第八阶段结论先行

第八阶段不再解决“系统主结构是否成立”的问题，而是开始解决：

- generation contract 能不能彻底压成 V1 统一事实来源
- runtime 例外分支能不能继续被清掉
- 版本流和专业版的高频使用链能不能再压短一截
- 双模式异常态、空态、恢复态、降级态能不能进入最终统一
- 整个工作台能不能进入上线前最终回归与验收准备

一句话总结：

第七阶段让系统更像可上线的 V1，第八阶段开始做 V1 上线前的最终封口与验收准备。

## 2. 为什么现在进入第八阶段

经过前七阶段之后，当前已经具备以下前提：

- contract 已经非常接近统一事实来源
- runtime 分支控制已经相对稳定
- 版本流已经具备较强的高频消费能力
- 专业版已经开始像连续决策工作台
- 双模式和模板入口已经进入明显收边阶段

因此，当前最合理的推进方向，不是继续扩展，而是做最后一轮：

- 封口
- 清例外
- 补异常态
- 补验收准备

## 3. 第八阶段总目标

第八阶段总目标可以收口成 3 句：

1. 让主干结构真正进入 V1 最终收口状态
2. 让高频主流程和高频决策链再压短、再稳定
3. 让系统进入上线前最终回归和验收准备阶段

## 4. 第八阶段产品原则

- 优先封口，不优先扩展
- 优先统一，不优先再造新层
- 优先异常态和恢复态稳定，不优先继续增加能力外延
- 优先回归和验收准备，不优先再造更多新入口
- 优先 V1 最终可上线，不优先远期平台能力

## 5. 第八阶段范围总览

建议把第八阶段收口成 5 个方向。

| 方向 | 名称 | 目标 | 优先级 |
|---|---|---|---|
| P81 | Contract 最终封口 | 让 generation contract 更接近 V1 最终唯一事实来源 | P0 |
| P82 | Runtime 例外分支清理 | 让模板 runtime 进一步减少特例和旁路 | P0 |
| P83 | 版本流最后一轮提效 | 让继续、重试、分叉的高频链更短 | P0 |
| P84 | 专业版连续决策收边 | 让专业版高频决策链更紧凑、更稳定 | P1 |
| P85 | 上线前回归与验收准备 | 让双模式、模板入口、回流链的异常态和回归准备更完整 | P1 |

## 6. P81 Contract 最终封口

### 6.1 目标

让 generation contract 更接近 V1 最终唯一事实来源。

### 6.2 当前问题

- 仍有少量桥接和 fallback 痕迹
- 个别链路对 contract 的消费还不是完全默认

### 6.3 第八阶段内建议做

- 继续清理残余 bridge / fallback 语义
- 强化关键主流程默认围绕 contract 运转
- 为最终回归和验收建立更清楚的 contract 主干判断

## 7. P82 Runtime 例外分支清理

### 7.1 目标

让模板 runtime 进一步减少特例和旁路逻辑。

### 7.2 当前问题

- 个别入口和边界态仍有例外处理
- runtime 与异常态、恢复态的统一还可继续压

### 7.3 第八阶段内建议做

- 清理低频入口上的 runtime 例外逻辑
- 强化 runtime 与降级、恢复、继续动作的一致性
- 让普通版、模板入口、结果动作回流更统一

## 8. P83 版本流最后一轮提效

### 8.1 目标

让继续、重试、分叉的高频链再更短一点。

### 8.2 当前问题

- 复杂多轮链路下仍需一定摘要阅读成本
- 高级用户在多版派生时仍有少量判断负担

### 8.3 第八阶段内建议做

- 继续压缩版本差异消费路径
- 强化高频动作判断提示
- 优化 Works / Tasks / Studio 三处之间的高频回流效率

## 9. P84 专业版连续决策收边

### 9.1 目标

让专业版高频决策链更紧凑、更稳定。

### 9.2 当前问题

- 连续派生、重跑、对照、校准之间仍有少量分散感
- 高强度使用下还可以更顺一点

### 9.3 第八阶段内建议做

- 继续压短连续派生与连续重跑路径
- 优化参数、Prompt、执行信息的决策消费顺序
- 强化来源版 / 当前版 / 目标版之间的直达关系

## 10. P85 上线前回归与验收准备

### 10.1 目标

让双模式、模板入口、回流链的异常态和回归准备更完整。

### 10.2 当前问题

- 主流程虽已稳定，但异常态和验收说明还可继续统一
- 上线前还需要更明确的回归视角

### 10.3 第八阶段内建议做

- 补齐关键异常态、空态、恢复态、降级态
- 为手工验收与发布前回归整理更稳状态
- 统一关键主流程的状态提示和恢复反馈

## 11. 第八阶段最值得优先做的 6 个任务

1. 清掉残余 contract bridge / fallback
2. 清掉残余 runtime 特例分支
3. 再压短版本流高频操作链
4. 再压短专业版连续决策链
5. 补齐关键异常态和恢复态
6. 为上线前回归与最终验收准备更稳基础

## 12. 第八阶段明确不做

- 不做新大型功能模块
- 不做团队协作
- 不做项目空间
- 不做开放 API 平台
- 不做模板 CMS
- 不做 Skill 市场
- 不做企业审批流

## 13. 第八阶段成功标准

- contract 更接近 V1 最终唯一事实来源
- runtime 特例继续减少
- 版本高频链更短
- 专业版连续决策链更顺
- 双模式与模板入口的异常态和验收准备更完整

## 14. 第八阶段总结

第八阶段不是新扩展轮，而是 V1 上线前的最终封口与验收准备轮。

一句话收口：

第七阶段让系统更像可上线产品，第八阶段要让它更像“可以放心上线的产品”。
