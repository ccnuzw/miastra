# 图像 Skill 平台 V1 · 第九阶段最终验收与上线执行清单

## 文档说明

- 文档目标：把第九阶段需要执行的手工验收、回归重点和上线执行准备收成一份可直接照着做的清单。
- 文档定位：承接 [image-skill-platform-phase9-scope-v1.md](./image-skill-platform-phase9-scope-v1.md) 与 [image-skill-platform-phase9-backlog-v1.md](./image-skill-platform-phase9-backlog-v1.md) 中 E95 的最终执行文档。
- 当前状态：第九阶段最终验收与上线执行准备文档。
- 更新时间：2026-05-05。

## 1. 结论先行

第九阶段的重点不是“再证明功能很多”，而是“确认最终主链已经足够稳定，团队可以按同一套步骤做回归、做验收、做上线前准备”。

这份文档只解决 3 件事：

1. 手工验收按什么顺序做
2. 最后一轮回归最该盯什么
3. 上线执行前后要准备什么

一句话收口：

第九阶段要把双模式、模板入口、版本回流和专业版连续链，收成一份可直接执行的上线前清单。

## 2. 本轮回归总原则

- 优先验证主链，不优先扩大样例数量
- 优先验证入口一致性，不优先验证外围展示细节
- 优先验证恢复态、异常态、降级态，不优先继续抠新体验
- 优先验证“能否继续下一步”，不优先验证“文案是否最优雅”
- 优先验证双模式共享主干，不接受同一来源在不同入口出现明显行为分叉

## 3. 回归前置条件

### 3.1 环境要求

- 已完成登录，可访问 `/app/studio`、`/app/templates`、`/app/tasks`、`/app/works`
- Provider 配置可用，至少能完成一轮真实或预发布环境的生成
- 模板库可读写
- 任务列表和作品列表可正常刷新

### 3.2 建议准备的数据

建议在开始前准备以下最小数据集：

- 至少 1 个适合普通版起手的模板
- 至少 1 个适合专业版起手的模板
- 至少 1 个带参考图的图生图样例
- 至少 1 个可以形成版本回流的已有作品
- 至少 1 个可在任务页看到重试或复跑信息的任务批次

### 3.3 建议先执行的自动化门禁

```bash
npm run release:check
```

如果这条命令没有通过，本轮不应进入上线准备确认。

## 4. 最终手工验收视角

这一轮建议固定按 6 组视角执行，不要来回跳。

### 4.1 视角 A：普通版最短起手链

目标：

- 验证普通版能从无基线状态进入第一版
- 验证第一版结果能自然接到下一步动作

执行路径：

1. 打开 `/app/studio`
2. 保持普通版模式
3. 在没有模板、没有来源版回流的情况下直接输入一句需求
4. 发起一次生成
5. 在结果区执行一次继续修改或结果动作回流

预期结果：

- 工作台允许直接起手，不要求先补模板
- 第一版结果生成后，可以继续把当前结果带回输入区
- 普通版提示聚焦“先出一版再继续改”，不出现专业版控制链式阻碍

### 4.2 视角 B：模板入口到工作台

目标：

- 验证模板页不只是存 Prompt，而是真正能作为 Skill 入口
- 验证模板推荐入口与工作台模式切换一致

执行路径：

1. 打开 `/app/templates`
2. 任选 1 个推荐普通版起手模板进入普通版
3. 再任选 1 个推荐专业版起手模板进入专业版
4. 回到工作台后分别检查模板上下文、场景字段和入口提示

预期结果：

- 模板卡片能清楚区分普通版起手或专业版起手
- 从模板进入后，工作台会恢复对应模板上下文
- 若模板更适合另一种模式，入口会自动按推荐路径纠偏

### 4.3 视角 C：版本回流主链

目标：

- 验证作品页和任务页都能把来源版带回工作台
- 验证“继续这一版 / 分叉这一版 / 同基线重跑”判断一致

执行路径：

1. 打开 `/app/works`
2. 从一张已有作品执行一次“继续这一版”
3. 从同一张或另一张作品执行一次“分叉 / 重跑这一版”
4. 打开 `/app/tasks`
5. 从一个已完成任务再次回流到工作台

预期结果：

- 作品页与任务页的版本来源语义一致
- 工作台恢复的 Prompt、参数和来源提示能对上
- 当参考图缺失时，会明确提示先补参考图，不会假装已完全恢复

### 4.4 视角 D：专业版连续链

目标：

- 验证专业版可以连续完成校准、对照、执行和回流
- 验证专业版不是一次性展示页，而是真能连着做决策

执行路径：

1. 打开 `/app/studio`
2. 切到专业版
3. 从模板或来源版进入专业版
4. 完成一次 Prompt 校准
5. 调整一次参数或执行基线
6. 发起生成后，再从结果回流一次

预期结果：

- 专业版能连续承接模板基线或来源版基线
- Prompt、参数、执行判断三类信息不会明显脱节
- 回流后能继续判断是同基线重跑还是形成派生

### 4.5 视角 E：异常态与恢复态

目标：

- 验证最后一轮回归不是只看成功态
- 验证关键异常不会把人带进假成功或假恢复

至少确认以下场景：

- Provider 仍在恢复中时，工作台提示可先整理 Prompt、模板或参考图
- 模板库读取失败时，模板入口与模板回流提示明确
- 作品区未刷新到最新状态时，会提示先刷新再做回流验收
- 任务结果已返回但作品映射未稳定时，会提示先刷新确认落盘
- 参考图只恢复部分内容时，会提示先补齐再重跑

### 4.6 视角 F：上线前最小闭环

目标：

- 用一轮最小真实业务闭环确认系统已进入可上线执行准备状态

建议闭环：

1. 模板页进入工作台
2. 在普通版或专业版完成一轮生成
3. 到任务页确认状态流转
4. 到作品页确认结果落盘
5. 从作品页回流继续一次
6. 完成一次导出或批量检查

预期结果：

- 模板入口、生成链、任务链、作品链、回流链能闭合
- 页面之间没有明显断链或状态失真

## 5. 最终回归重点

最后一轮不要平均用力，优先盯以下 4 组重点。

### 5.1 双模式一致性

重点确认：

- 普通版与专业版共享同一来源版时，恢复语义是否一致
- 同一个模板从不同模式进入时，场景与模板上下文是否一致
- 同一个结果动作回流时，是否只是呈现层不同，而不是主干判断不同

### 5.2 模板入口一致性

重点确认：

- 模板推荐入口是否与实际进入模式一致
- 模板默认追问、结构字段和后续动作提示是否仍然对齐
- 模板库异常时，是否给出明确恢复动作

### 5.3 版本回流与恢复态

重点确认：

- 作品页回流与任务页回流是否共享同一套恢复语义
- 参考图恢复不完整时，是否明确阻止误判为“可直接同版重跑”
- 预览图未恢复、结果映射未完成时，是否仍允许安全地继续参数回流

### 5.4 专业版连续链

重点确认：

- 专业版从模板进入后的第一步是否清楚
- 连续派生、连续重跑、结果回流后的下一步是否清楚
- Prompt、参数、执行判断三块是否仍维持可连续消费顺序

## 6. 建议执行顺序

建议固定按以下顺序执行最后一轮手工回归：

1. `/app/templates`
2. `/app/studio` 普通版最短链
3. `/app/studio` 专业版连续链
4. `/app/tasks`
5. `/app/works`
6. 再回 `/app/studio` 做版本回流闭环

原因：

- 先验证模板入口
- 再验证工作台主链
- 再验证任务与作品回流
- 最后验证完整闭环

## 7. 上线执行准备

### 7.1 上线前必须明确的责任分工

至少明确以下角色：

- 谁执行 `npm run release:check`
- 谁做手工验收记录
- 谁负责生产环境变量复核
- 谁负责发布后健康检查
- 谁负责发布后 30 分钟内观察任务、作品和 Provider 反馈

### 7.2 上线前检查项

- 确认最新目标分支为 `feature/image-skill-platform-v1`
- 确认本轮第九阶段相关改动已合入
- 确认 [release-regression.md](./release-regression.md) 自动化门禁已通过
- 确认本清单第 4 节的 6 组视角已至少走完 1 轮
- 确认模板页、工作台、任务页、作品页可访问
- 确认 `/health` 与 `/health/store` 正常

### 7.3 上线当天建议执行顺序

1. 执行自动化门禁
2. 按本清单完成最终手工验收
3. 复核环境变量与部署入口路由
4. 执行部署
5. 做发布后健康检查
6. 做发布后最小业务闭环

### 7.4 发布后 30 分钟观察点

- 新任务是否能正常进入 `queued -> running -> succeeded / failed`
- 作品是否能稳定落盘并出现在 `/app/works`
- 从任务页、作品页回流到工作台是否仍正常
- 模板入口是否还能正常进入普通版与专业版
- 是否出现明显的 Provider 恢复异常、模板库读取异常或作品同步异常

## 8. 异常时的决策建议

### 8.1 可以带风险上线但要记录的问题

- 非主链文案轻微不一致
- 边缘空态信息不够理想，但不影响继续操作
- 个别统计数字或推荐提示略保守，但不影响主干判断

### 8.2 不应继续上线的问题

- 模板入口无法稳定进入工作台
- 普通版无法完成“输入一句需求 -> 第一版 -> 结果回流”最短链
- 专业版无法完成“基线进入 -> 生成 -> 回流继续”连续链
- 任务页和作品页对同一来源版给出互相冲突的回流语义
- 参考图缺失时仍误导用户直接同基线重跑

## 9. 验收记录模板

建议执行时直接记录以下信息：

| 项目 | 记录内容 |
|---|---|
| 执行时间 | 例如 `2026-05-05 18:30` |
| 执行环境 | 本地 / 预发布 / 生产 |
| 执行人 | 姓名或角色 |
| 自动化结果 | `release:check passed / failed` |
| 模板入口 | 通过 / 不通过 |
| 普通版最短链 | 通过 / 不通过 |
| 专业版连续链 | 通过 / 不通过 |
| 任务回流 | 通过 / 不通过 |
| 作品回流 | 通过 / 不通过 |
| 异常态与恢复态 | 通过 / 不通过 |
| 阻断问题 | 无 / 具体问题 |
| 是否允许上线 | 是 / 否 |

## 10. 总结

第九阶段的最终验收，不是再证明系统有多少功能，而是确认以下四条已经能稳定闭合：

- 双模式
- 模板入口
- 版本回流
- 专业版连续链

只要这四条在最后一轮回归里都能按同一套步骤稳定执行，Miastra 图像 Skill 平台 V1 就算真正进入上线执行准备状态。
