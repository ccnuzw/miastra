# 图像 Skill 平台 V1 · 普通版与专业版完整产品地图

## 文档说明

- 文档目标：把普通用户版与专业版收拢成一张完整产品地图，明确两种前台模式如何共存、共享和分工。
- 文档定位：承接 [image-skill-platform-v1-prd.md](./image-skill-platform-v1-prd.md)、[image-skill-platform-consumer-v1.md](./image-skill-platform-consumer-v1.md)、[image-skill-platform-pro-v1-prd.md](./image-skill-platform-pro-v1-prd.md) 与 [image-skill-platform-mode-switching-v1.md](./image-skill-platform-mode-switching-v1.md) 的产品总图文档。
- 当前状态：产品地图主文档。
- 更新时间：2026-05-05。

## 1. 文档结论先行

Miastra 的前台不应被拆成“两套彼此割裂的产品”，而应被定义成：

- 一套统一的图像 Skill 平台
- 两种不同表达层
- 一条共享任务与资产主链路

一句话总结：

普通版负责让用户快速开工，专业版负责让用户深度控制，而底层任务、模板、作品、历史、Provider、额度和版本链必须保持统一。

## 2. 为什么必须先做完整产品地图

如果只分别写普通版和专业版文档，容易出现三个问题：

- 页面结构各自合理，但放到一起后导航和路径冲突。
- 普通版和专业版各自有工作台，却没有统一的任务与资产主线。
- 开发时不知道哪些页面是新建，哪些页面是重构，哪些能力应共享。

因此在正式进入开发实施前，必须先回答这三个问题：

1. 用户从哪里开始进入产品。
2. 普通版和专业版分别承接哪个阶段的任务。
3. 哪些能力是前台模式差异，哪些能力是底层统一资产。

## 3. 产品地图总原则

- 一个产品，不是两套孤立站点。
- 一个任务系统，不是普通版一套任务、专业版一套任务。
- 一个作品与历史资产池，不是各存各的。
- 一个 Provider 与额度体系，不是普通版和专业版各管各的。
- 一个版本链，不是普通版结果页和专业版结果页断开。
- 首页优先服务普通用户，专业用户从创作页开始放大控制力。

## 4. 完整产品地图总览

可以把整个产品理解为 7 层。

| 层级 | 主要作用 | 普通版角色 | 专业版角色 | 底层共享能力 |
|---|---|---|---|---|
| L0 应用壳层 | 登录、导航、模式记忆、账号与权限 | 默认入口与轻导航 | 进入控制视图与高级入口 | 认证、用户资料、角色、模式偏好 |
| L1 任务入口层 | 帮用户开始任务 | 强引导、强场景、低门槛 | 弱化，但仍可快速进工作台 | 模板推荐、最近任务、常用入口 |
| L2 创作工作台层 | 发起生成与编辑 | 引导式工作台 | 控制式工作台 | Skill Engine、任务创建、上传、参数、Provider 解析 |
| L3 结果与继续创作层 | 看结果并继续改 | 继续改动作优先 | Prompt、参数、版本、重跑优先 | 结果图、任务状态、版本父子关系 |
| L4 资产与复用层 | 管理作品、历史、模板 | 找回与继续 | 筛选、对比、复用、导出 | 作品库、历史、模板、批次、标签 |
| L5 执行与设置层 | 管理模型、Provider、计费和运行策略 | 默认隐藏，仅轻量提示 | 可见可控 | Provider 配置、模型选择、Billing、运行模式 |
| L6 运营与后台支撑层 | 管理用户、任务、Provider、审计 | 无直接前台暴露 | 无直接前台暴露 | 后台管理、任务运营、Provider 运营 |

## 5. 各层详细展开

## 5.1 L0 应用壳层

这一层解决“用户如何进入系统、如何记住模式、如何在不同页面间流转”。

当前仓库对应：

- [AppShell.tsx](/Users/apple/Progame/newfeng/miastra/src/layouts/AppShell.tsx)
- [AppRouter.tsx](/Users/apple/Progame/newfeng/miastra/src/routes/AppRouter.tsx)

这一层的产品要求：

- 新用户默认进入普通版视角。
- 老用户记住上次使用的是普通版还是专业版。
- 模式切换不等于换账号或换任务上下文。
- 顶部导航始终基于统一产品，而不是显示为两套站点。

推荐前台统一导航：

- 工作台
- 模板
- 作品
- 任务
- 账户
- 配置中心
- 支付
- 后台

其中：

- 普通用户主要感知 `工作台 / 作品 / 任务`
- 专业用户主要感知 `工作台 / 模板 / 作品 / 配置中心`

## 5.2 L1 任务入口层

这一层是普通版的主战场，但它属于整个产品地图，而不是“普通版专属孤岛”。

主要目标：

- 让新用户快速开始
- 让回访用户快速继续
- 让专业用户也能快速从高频模板或历史切入

普通版呈现：

- 你现在想做什么图
- 从文字生成图片
- 按现有图片继续修改
- 做商品图
- 做海报或宣传图
- 做头像或人像图
- 继续上次创作

专业版呈现：

- 可以保留同一个入口页
- 但进入创作页后更快切换到控制视图
- 专业用户不需要单独一张“高级首页”

底层共享：

- 模板推荐逻辑
- 最近任务
- 最近作品
- 最近使用的模板与参数

## 5.3 L2 创作工作台层

这一层是整个产品的核心。

普通版工作台：

- 按任务语言组织
- 一句话需求
- 上传图
- 轻量偏好
- 帮我补全
- 开始生成

专业版工作台：

- 模板可见
- Prompt 可见
- 参数可见
- Provider 可见
- 结果重跑可见

两种视图的本质区别不在于是否能生成，而在于输入面板的组织方式不同。

共享底层能力：

- 统一任务创建接口
- 统一上传与参考图机制
- 统一生成状态
- 统一结果回流
- 统一版本父子关系

当前仓库对应：

- [StudioPage.tsx](/Users/apple/Progame/newfeng/miastra/src/pages/app/StudioPage.tsx)
- [StudioEditorColumn.tsx](/Users/apple/Progame/newfeng/miastra/src/pages/app/studio/StudioEditorColumn.tsx)
- [StudioGenerationColumn.tsx](/Users/apple/Progame/newfeng/miastra/src/pages/app/studio/StudioGenerationColumn.tsx)
- [StudioWorksColumn.tsx](/Users/apple/Progame/newfeng/miastra/src/pages/app/studio/StudioWorksColumn.tsx)

## 5.4 L3 结果与继续创作层

这一层不是工作台附属区，而是决定产品留存的关键层。

普通版结果层关注：

- 看结果
- 更像这一张
- 换个风格
- 只改这一部分
- 保留主体换背景
- 再生成几张

专业版结果层关注：

- 查看最终 Prompt
- 编辑 Prompt 再跑
- 复制参数重跑
- 切换模板变体
- 查看版本关系
- 导出参数与 Prompt

共享底层能力：

- 统一结果对象
- 统一候选图集合
- 统一任务参数快照
- 统一版本链

## 5.5 L4 资产与复用层

这一层统一承接模板、作品、历史和任务，不能拆成多个孤立列表。

普通版看重：

- 找回做过的图
- 继续做
- 收藏和下载

专业版看重：

- 按模板筛选
- 按模型筛选
- 按参数筛选
- 按任务或批次回看
- 复制参数重跑

当前仓库已有页面可以承接：

- [TemplatesPage.tsx](/Users/apple/Progame/newfeng/miastra/src/pages/app/TemplatesPage.tsx)
- [WorksPage.tsx](/Users/apple/Progame/newfeng/miastra/src/pages/app/WorksPage.tsx)
- [TasksPage.tsx](/Users/apple/Progame/newfeng/miastra/src/pages/app/TasksPage.tsx)

V1 的关键不是新建更多资产页，而是让这三页之间建立统一语义：

- 模板资产
- 结果资产
- 任务资产

## 5.6 L5 执行与设置层

这一层解决的是“图是如何被执行出来的”。

普通版原则：

- 默认不暴露 Provider 和模型细节
- 只给轻量提示，如已连接、当前可用、预计消耗

专业版原则：

- 可查看当前 Provider
- 可切换默认模型
- 可切换平台托管或用户自带 Provider
- 可理解执行失败后的降级路径

当前仓库对应：

- [ProviderConfigPage.tsx](/Users/apple/Progame/newfeng/miastra/src/pages/app/ProviderConfigPage.tsx)
- [BillingPage.tsx](/Users/apple/Progame/newfeng/miastra/src/pages/app/BillingPage.tsx)

## 5.7 L6 运营与后台支撑层

这一层不属于普通版或专业版前台的一部分，但必须放进完整产品地图。

原因：

- 普通版和专业版的体验最终都依赖任务状态、Provider 健康度、额度、公共模型、后台运营能力。
- 没有后台支撑，前台的“简单体验”无法长期稳定。

当前仓库对应：

- `src/pages/app/admin/*`
- `server/src/admin/*`
- `server/src/provider-config/*`
- `server/src/generation-tasks/*`

## 6. 完整页面地图

下面这张表把“产品地图”落到当前真实页面上。

| 页面 | 当前路由 | 在完整产品地图中的角色 | 普通版优先级 | 专业版优先级 | 备注 |
|---|---|---|---|---|---|
| 工作台 | `/app/studio` | 核心创作页、结果页、模式切换主入口 | 最高 | 最高 | 不建议拆成两套路由，建议保留统一入口 |
| 模板 | `/app/templates` | 模板资产页 | 中 | 高 | 后续逐步从 Prompt 模板升级为 Skill 模板视图 |
| 作品 | `/app/works` | 结果资产页 | 高 | 高 | 普通版偏继续做，专业版偏筛选和复用 |
| 任务 | `/app/tasks` | 任务状态与历史页 | 中 | 高 | 专业版更依赖这里 |
| 配置中心 | `/app/providers` | Provider 与执行配置页 | 低 | 高 | 普通版轻提示，专业版深控制 |
| 支付 | `/app/billing` | Billing 与额度页 | 中 | 中 | 两种模式共享 |
| 后台 | `/app/admin/*` | 运营支撑层 | 无 | 无 | 但对前台稳定性关键 |

## 7. 三条核心用户主链路

## 7.1 普通版主链路

首页任务入口 -> 引导式创作页 -> 结果页继续改 -> 保存到作品 -> 历史中再次打开

这是面向大多数用户的主线，必须最短、最顺。

## 7.2 专业版主链路

统一工作台 -> 模板/Prompt/参数控制 -> 结果页查看与重跑 -> 作品或任务中复用 -> 继续版本链迭代

这是面向高频用户的主线，必须最可控、最可追溯。

## 7.3 共享复用主链路

一次生成任务 -> 写入任务 -> 写入结果 -> 写入作品 -> 关联模板/参数/Provider -> 历史中再次打开

这是整个平台真正的资产主线，也是普通版和专业版能够共存的基础。

## 8. 第一阶段产品收口

完整产品地图虽然覆盖很多层，但 V1 前台开发不应同时铺满所有页面。

第一阶段最值得优先打通的是：

- `/app/studio`
- `/app/works`
- `/app/tasks`

原因：

- `studio` 负责开始和继续创作
- `works` 负责结果回流与二次创作
- `tasks` 负责任务状态、失败恢复和专业追踪

模板、Provider、Billing 保持现有页面，但先做最小必要升级。

## 9. 这张产品地图的开发意义

这张地图为后续开发提供了三个明确约束：

1. 不要把普通版和专业版开发成两套独立前台。
2. 不要为了普通版去阉割专业资产，也不要为了专业版去污染普通流程。
3. 第一阶段要优先打通共享任务链路，而不是先做很多新页面。

一句话总结：

产品层看起来是双模式，系统层必须始终是一张图。
