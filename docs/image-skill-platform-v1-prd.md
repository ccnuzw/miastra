# 图像 Skill 平台 V1 PRD

## 文档说明

- 文档目标：定义 Miastra 下一阶段从“图片工作台”升级为“图像 Skill 平台”的产品方向与一期交付范围。
- 当前状态：产品规划主文档。
- 更新时间：2026-05-05。

术语修正说明：

- 本文统一使用“文生图”表达通用的文本生成图片能力。
- 若后续需要支持特定垂直场景，应作为独立模板分类或独立 Skill 处理。

## 1. 产品定义

### 1.1 一句话定义

一个面向 Web 用户的图像创作平台，通过 `Skill Engine + 模板系统 + 动态追问 + Prompt 渲染 + 生成/编辑编排 + 资产归档`，让用户更稳定地完成从想法到结果图的全过程。

### 1.2 产品本质

- 前台表现层是统一创作工作台。
- 中台核心是图像 Skill 平台。
- 竞争力重点不在页面数量，而在图像能力的结构化、可复用和可沉淀。

### 1.3 产品不是

- 纯文生图页面
- 工具集合站
- 只有上传与生成按钮的轻量 Demo
- 只服务专业 Prompt 工程师的高级面板

### 1.4 产品应该是

- 对新手友好
- 对高频用户可复用
- 对运营可配置
- 对未来多模型、多 Provider、商业化和批量能力有扩展空间

## 2. 背景与问题

当前工作台存在的核心问题，不是 UI 不够多，而是底层能力没有系统化：

- 功能以单点页面形态存在，缺少统一能力编排。
- 文生图、图生图、局部编辑、模板、历史之间关系松散。
- 用户每次都需要重新试错，缺少模板与参数复用。
- 历史结果更像图片列表，而不是可继续创作的资产。
- Prompt 没有被系统化沉淀为结构化能力。
- 页面入口与底层能力之间缺少稳定映射。

参考内容带来的关键启发不是“界面长什么样”，而是以下能力方法论：

- 同一个 Skill 能在不同运行环境中优雅降级。
- 模板不是示例文案，而是结构化能力单元。
- 缺失信息应按模板定义的顺序精准追问。
- Prompt 和结果都必须归档，才能真正复用。
- 生成与编辑应统一在一个能力系统里，而不是多个割裂工具。

## 3. 目标与边界

### 3.1 业务目标

- 把图片创作从“试一试”升级为“稳定完成任务”。
- 把单次生成升级为可复用、可继续创作的工作流。
- 为未来模板增长、模型切换、商业化和批量化打底。

### 3.2 用户目标

- 快速开始，不被空白页卡住。
- 不懂 Prompt 也能得到可用结果。
- 能基于已有结果继续修改，而不是每次重来。
- 能复用过去的成功经验。
- 在模型不可用时，仍然至少拿到高质量可执行 Prompt。

### 3.3 平台目标

- 把图片能力沉淀为可管理的 Skill 资产。
- 把模板、参数、Prompt、结果和版本关系系统化。
- 为后续模板市场、运营配置、BYOK、批量任务预留底盘。

### 3.4 V1 不追求

- 团队协作体系
- 企业级审批流
- 大规模自动化工作流
- 对外开放 API 平台
- 模板生态市场化闭环

## 4. 核心设计原则

- 平台优先于页面
- 模板优先于空白输入
- 追问优先于大表单
- 结果继续创作优先于一次性出图
- Prompt 归档优先于临时生成
- 统一编排优先于散装工具页
- 可降级优先于直接报错失败

## 5. 目标用户

### 5.1 新手用户

需要快速产出海报图、商品图、头像图、封面图、文生图结果，不想研究复杂提示词。

### 5.2 改图用户

已经有原图或参考图，希望做图生图、换背景、去杂物、局部修改、局部重绘、扩图。

### 5.3 高频用户

希望沉淀模板、复用参数、回看历史、稳定重复出图。

### 5.4 运营或半专业用户

希望通过预设模板快速完成高频业务素材。

## 6. 核心概念

- `Skill`：一类图像能力集合，例如商品图 Skill、海报 Skill、背景替换 Skill。
- `Template`：Skill 下的具体模板，包含结构、参数、提问顺序、默认值、变体和约束。
- `Argument`：模板参数项，可分为必填、可默认、可随机、可自动补全。
- `Question Flow`：模板驱动的动态追问流程。
- `Prompt Draft`：渲染前的结构化输入草稿。
- `Rendered Prompt`：最终可执行 Prompt。
- `Run Mode`：当前任务的执行模式。
- `Task`：一次生成或编辑任务。
- `Asset`：原图、参考图、蒙版图、结果图、Prompt 文件等资产。
- `Version Node`：某张结果图在版本链中的节点。
- `Provider Profile`：平台托管或用户自带的模型配置。

## 7. Web 版运行模式

CLI 的 A/B/C 模式思想必须保留，但要翻译成 Web 产品形态。

### 7.1 W1 平台托管模式

- 平台负责调用模型、执行任务、返回图片结果。
- 默认优先进入该模式。

### 7.2 W2 用户自带 Provider 模式

- 用户配置自己的 key、base URL、model。
- 平台负责模板、追问、Prompt 渲染、任务归档。

### 7.3 W3 Prompt 顾问模式

- 当模型不可用、额度不足、用户只想拿走 Prompt 时启用。
- 平台输出高质量 Prompt 包和执行建议。
- 不伪装成已生成成功。

### 7.4 模式规则

- 默认优先 W1。
- 若用户配置自带 Provider，可显式切换或自动走 W2。
- 当 W1/W2 不可执行时，降级到 W3。
- 所有模式都保留 Prompt 归档。
- W1/W2 产出图片，W3 至少产出 Prompt 资产。

## 9. Skill Engine

Skill Engine 是整个平台的核心调度层。

### 9.1 核心职责

- 识别用户当前意图
- 映射到合适 Skill
- 选择合适模板
- 发起动态追问
- 收集参数并生成 Prompt Draft
- 渲染最终 Prompt
- 选择执行模式与 Provider
- 发起生成或编辑任务
- 保存任务过程与资产
- 生成版本关系
- 在失败时自动降级

### 9.2 产品要求

- 前台所有创作入口都必须经由 Skill Engine。
- 不能让页面直接拼接 Prompt 后裸调接口。
- Skill Engine 必须保留足够上下文，支持后续继续创作。

## 10. 模板系统

模板系统不是辅助功能，而是平台主资产。

### 10.1 模板组织结构

- 一级：Skill 分类
- 二级：具体模板
- 三级：模板变体

### 10.2 V1 首批分类建议

- 文生图
- 图生图
- 商品图
- 海报图
- 头像与人像图
- 白底电商图
- 背景替换
- 局部替换
- 去杂物 / 去文字
- UI 样机

### 10.3 每个模板必须包含

- 模板名称
- 所属 Skill
- 适用范围
- 何时使用
- 不适用边界
- 缺失信息优先提问顺序
- 主模板结构
- 参数定义
- 默认值策略
- 自动补全策略
- 可选变体
- 避免事项
- 示例结果
- 推荐模型与模式

## 12. 动态追问系统

动态追问系统是 V1 体验差异化的关键。

### 12.1 设计目标

- 不让用户面对一个巨大的静态表单。
- 不笼统问“你想要什么风格”。
- 按模板需要逐步问最关键的信息。
- 支持“自动补全次要信息”。

### 12.2 追问规则

- 每个模板定义自己的问题优先级。
- 必填项优先问。
- 高影响字段优先问。
- 次要字段可提供默认或自动模式。
- 用户可选择“快速模式”跳过次要项。
- 用户随时可回头修改答案。

### 12.3 产品要求

- 动态追问不是聊天机器人。
- 动态追问的本质是参数补全器。
- 问题必须围绕模板字段组织，而不是自由发散。

## 13. Prompt 渲染与归档

Prompt 必须从一次性输入，升级为正式资产。

### 13.1 每次任务都应保存

- 用户原始需求
- 模板 ID 与模板版本
- 追问回答
- 自动补全内容
- 最终渲染 Prompt
- 执行模式
- 模型参数

### 13.2 Prompt 输出建议

- 用户可读的自然语言 Prompt
- 系统可读的结构化 Prompt 快照

### 13.3 归档要求

- 所有任务必须有 Prompt 记录。
- 历史中必须能查看 Prompt。
- 用户可复制、再次编辑、导出。
- 从历史继续创作时，默认带回 Prompt 资产。

## 14. Provider 路由与执行系统

### 14.1 Provider 层必须支持

- 平台默认 Provider
- 用户自带 OpenAI 兼容 Provider
- 按 Skill 或模板推荐默认模型
- 执行失败时降级
- 记录每次执行使用的 provider/model

### 14.2 执行类型

- `generate`
- `edit`
- `inpaint`
- `outpaint`
- `reference-guided edit`

### 14.3 执行前必须展示

- 预计消耗
- 预计耗时
- 当前运行模式
- 当前 Provider / model

### 14.4 执行失败要求

- 优先提示重试
- 可切换 Provider
- 可降级到 Prompt 顾问模式
- 不允许用户丢失已经完成的参数收集

## 15. 资产归档与版本链

V1 必须把“结果列表”升级成“创作资产系统”。

### 15.1 每次任务归档的资产

- 原图
- 参考图
- 蒙版图
- Prompt Draft
- Rendered Prompt
- 结果图
- 参数快照
- Provider 快照
- 状态与错误信息

### 15.2 版本链规则

- 每个结果节点都可标记父节点。
- 用户可以从任一节点继续编辑。
- 用户可以复制该节点参数发起新任务。
- 历史页必须能看出当前结果来自哪一次演变。

## 18. 核心用户流程

### 18.1 从模板开始

- 用户进入首页或模板中心
- 选择模板
- 系统发起动态追问
- 用户补全核心信息
- 系统渲染 Prompt
- 选择执行模式
- 发起任务
- 查看结果
- 继续创作或保存版本

### 18.2 已有图改图

- 用户选择编辑类 Skill
- 上传原图
- 系统根据模板问“保留什么、替换什么、是否重算阴影”等
- 渲染编辑 Prompt
- 发起任务
- 返回结果
- 继续局部修改或切换变体

### 18.3 失败降级

- 用户填写完参数
- 平台执行失败或模型不可用
- 系统给出切换 Provider 或降级为 Prompt 顾问
- 用户仍可拿走高质量 Prompt
- 历史中保留完整任务记录

### 18.4 从历史继续

- 用户进入历史版本页
- 打开某次结果
- 查看 Prompt 与参数
- 点击继续创作
- 自动带回模板、素材、参数和版本父节点
- 发起下一次任务

## 21. 数据模型建议

建议至少规划以下核心表：

- `skills`
- `templates`
- `template_versions`
- `template_arguments`
- `question_flows`
- `tasks`
- `task_runs`
- `task_assets`
- `rendered_prompts`
- `result_images`
- `version_nodes`
- `provider_profiles`

关键字段方向建议：

- `skills`：`name`、`category`、`description`、`enabled`
- `templates`：`skill_id`、`name`、`slug`、`scenario`、`status`
- `template_versions`：`template_id`、`dsl_schema`、`default_values`、`variant_meta`
- `question_flows`：`template_version_id`、`step_order`、`field_key`、`question_text`、`required`
- `tasks`：`user_id`、`skill_id`、`template_id`、`mode`、`status`
- `task_runs`：`task_id`、`provider_id`、`model`、`run_mode`、`error_message`、`cost`
- `task_assets`：`task_id`、`type`、`url`、`meta`
- `rendered_prompts`：`task_id`、`draft_json`、`final_text`、`final_structured`
- `version_nodes`：`task_id`、`parent_task_id`、`root_task_id`

## 23. 主要风险

### 23.1 风险一

只做出前台页面，没有 Skill Engine 底盘。

结果：

- 产品会退化为一个更复杂的生图页。

### 23.2 风险二

模板做成文档列表，没有执行能力。

结果：

- 模板无法真正驱动追问、渲染和复用。

### 23.3 风险三

追问系统做成聊天，而不是参数收集。

结果：

- 体验看似智能，实际效率低。

### 23.4 风险四

历史只存结果图，不存 Prompt 和模板上下文。

结果：

- 无法真正继续创作。

### 23.5 风险五

Provider 与运行模式没抽象好。

结果：

- 后续支持用户自带 key 或降级 Prompt 模式时，重构成本很高。

## 24. V1 验收标准

如果 V1 做对了，用户应该可以：

- 从首页或模板中心进入任意一个图像 Skill。
- 由系统按模板动态追问关键参数。
- 自动生成高质量 Prompt。
- 在平台托管或自带 Provider 下执行生成或编辑。
- 把 Prompt、素材、结果和版本关系全部沉淀下来。
- 在失败时退化为可直接执行的 Prompt 顾问模式。

## 25. 文档结论

当前路线不应再定义为“图片工作台 V1”，而应定义为“图像 Skill 平台 V1”。

两者差异在于：

- 前者重点是页面
- 后者重点是能力系统

工作台只是入口，Skill Engine 才是产品核心。
