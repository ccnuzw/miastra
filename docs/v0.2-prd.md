# Miastra Studio v0.2 PRD

## 0. 文档信息
- 日期：2026-05-01
- 基线版本：v0.1
- 状态：Draft

## 1. 产品概述
Miastra Studio 是面向个人创作者的 AI 图片创作工作台。v0.2 的核心目标，是把当前“能用的本地生图台”升级为“稳定、可复用、可管理的创作台”，让用户可以更高效地组织 Prompt、管理作品、控制批量任务，并可靠地导出与复用创作结果。

## 2. 背景与问题定义
v0.1 已经具备完整生图主流程，但在真实使用中仍存在以下问题：
- Prompt 和模板可用，但检索与复用效率不够高。
- 作品数量增多后，搜索、筛选、批量整理成本上升。
- 批量生成过程出现失败时，恢复和重跑成本偏高。
- 错误提示与状态表达不够标准，排查体验不稳定。
- 当前以本地持久化为主，工程稳定性和兼容性仍需补强。

## 3. 产品目标
1. 提升 Prompt 模板的沉淀、查找与复用效率。
2. 提升作品资产管理能力，让作品更容易找回、整理和复用。
3. 提升抽卡与队列工作流的稳定性、可控性与恢复能力。
4. 建立基础工程质量标准，为后续版本演进留出空间。

## 4. 非目标
v0.2 不解决以下问题：
- 登录 / 注册 / 账号体系
- 团队协作与权限管理
- 云端同步、对象存储与跨设备实时协同
- 社区、分享广场、内容分发
- 商业化收费、订阅、支付系统
- 完整后端业务平台建设
- 模型训练、模型托管与训练管理

## 5. 目标用户
- 个人视觉创作者
- 高频使用 AI 图片生成的重度用户
- 需要长期沉淀 Prompt、模板和作品资产的创作者
- 愿意配置 Provider 并对生成参数有一定理解的用户

## 6. 主要使用场景
1. 用户从模板库中快速找到常用 Prompt，直接发起创作。
2. 用户上传 1~4 张参考图，进行图生图或多参考图增强生成。
3. 用户一次发起多张抽卡任务，并在队列中暂停、恢复或取消。
4. 用户在作品库中通过搜索、标签和收藏快速整理结果。
5. 用户对失败任务进行重试，或者从批次中重新发起一轮生成。
6. 用户导出单图或整批作品，并保留 metadata 便于外部管理。

## 7. v0.2 范围定义
### 7.1 范围内
- Prompt 模板系统增强
- 作品资产管理增强
- 抽卡与队列工作流增强
- 错误提示与工程稳定性增强
- 现有导出、复用和持久化能力的补强

### 7.2 范围外

## 8. 功能需求

### Epic A：Prompt 模板系统增强
#### 目标
让用户可以更快找到、复用、编辑和整理 Prompt 模板，降低重复输入成本。

#### 功能需求
- 支持模板搜索。
- 支持模板分类与标签筛选。
- 支持模板编辑与复制为新模板。
- 支持模板最近使用排序。
- 支持模板在工作流中被快速调用。

#### 优先级
- P0：搜索、编辑复制
- P1：分类、标签
- P2：最近使用排序

#### 用户故事
- 作为创作者，我希望能快速搜索到常用模板。
- 作为创作者，我希望能把一个模板复制后再修改，而不影响原模板。
- 作为创作者，我希望能通过分类和标签管理不同风格的模板。

#### 验收标准
- 模板可按关键词正常检索。
- 复制后的模板与原模板相互独立。
- 分类和标签筛选结果准确。
- 最近使用的模板可以排到前面。

### Epic B：作品资产管理增强
#### 目标
让作品从“生成结果”变成“可整理、可检索、可复用的资产”。

#### 功能需求
- 支持作品批量标签。
- 支持作品搜索增强。
- 支持作品详情增强，展示参数快照、批次信息和相关元数据。
- 支持批量收藏与取消收藏。
- 支持更清晰的图片墙与列表联动体验。

#### 优先级
- P0：搜索增强、批量标签
- P1：详情增强、批量收藏

#### 用户故事
- 作为创作者，我希望能按标签快速找到某一批作品。
- 作为创作者，我希望在作品详情里直接看到当时的参数快照。
- 作为创作者，我希望能够一次给多张作品打标签或收藏。

#### 验收标准
- 搜索与筛选可覆盖主要查找路径。
- 批量标签操作结果可持久化。
- 作品详情能稳定展示关键参数与批次信息。
- 收藏状态在刷新后保持一致。

### Epic C：抽卡与队列工作流增强
#### 目标
让批量生成过程更可控、更可恢复，降低失败成本。

#### 功能需求
- 支持队列失败项一键重试。
- 支持批次视图增强，展示进度、状态与结果概览。
- 支持批次复跑入口，从历史批次快速再次发起。
- 支持错误提示标准化，统一反馈格式。
- 保留暂停、恢复、取消、重试等已有队列能力，并增强可见性。

#### 优先级
- P0：失败项重试、错误提示标准化
- P1：批次视图增强、批次复跑入口

#### 用户故事
- 作为创作者，我希望失败任务可以直接重试，而不用重建整批任务。
- 作为创作者，我希望能看到某一批任务的整体状态。
- 作为创作者，我希望能从历史批次快速再跑一次。

#### 验收标准
- 失败项可以独立重试，不影响其他成功任务。
- 批次状态、成功数、失败数可清晰展示。
- 从批次入口发起复跑后，任务参数保持可追溯。
- 错误提示统一且可理解。

### Epic D：工程质量与稳定性增强
#### 目标
保证 v0.2 在本地使用场景下稳定、可维护、可回归。

#### 功能需求
- 补齐基础测试与回归检查。
- 处理 parser、URL / 请求和存储兼容性问题。
- 保证 ZIP 导出与 metadata 结构稳定。
- 建立最小 Smoke Test 流程。
- 为后续后端代理与能力扩展预留接口。

#### 优先级
- P0：兼容性、导出、Smoke Test
- P1：预留接口与文档化

#### 用户故事
- 作为创作者，我希望升级版本后历史作品和模板还能正常打开。
- 作为创作者，我希望导出的 ZIP 和 metadata 能稳定使用。
- 作为维护者，我希望在发布前能快速做一次 Smoke Test。

#### 验收标准
- 构建持续可通过。
- 核心流程在 Smoke Test 中无阻塞问题。
- 兼容性变更不会破坏现有本地数据。
- 导出结构前后一致。

## 9. 信息架构与交互要求
- 主工作区应以“创作台 + 作品区 + 设置区”的结构组织。
- 模板库、作品库、队列状态应可快速切换。
- 关键操作应可从列表、详情与批次视图三处进入。
- 状态反馈应保持一致，包括成功、失败、进行中、暂停和已取消。
- 生成、收藏、标签、导出等高频操作应尽量减少跳转层级。

## 10. 数据要求
- Provider 配置继续本地保存。
- 模板、作品、批次、标签、收藏状态继续使用本地持久化。
- 作品数据需保留参数快照、时间戳、批次标识、参考图信息和导出信息。
- `metadata.json` 需与导出目录结构保持稳定。
- 若后续升级存储结构，需提供兼容与迁移策略。

## 11. 技术约束与依赖
- 当前仍以本地优先为前提，不依赖完整后端业务层。
- Provider 兼容 OpenAI Images API 形态，并支持 `/sub2api` 代理路径。
- 现有前端工程、Vite、Docker、Docker Compose、Nginx 代理链路需保持可用。
- 新增能力应尽量避免破坏现有 localStorage / IndexedDB 数据。
- 测试体系与工程化能力需与当前项目规模匹配，优先覆盖核心路径。

## 12. 指标与成功衡量
- 模板复用效率提升：用户更少重复手工输入。
- 作品找回效率提升：通过搜索、标签、收藏更快定位资产。
- 队列恢复效率提升：失败任务的重试成本降低。
- 导出稳定性提升：ZIP 与 metadata 导出成功率稳定。
- 工程稳定性提升：核心流程在回归检查中可稳定通过。

## 13. 版本计划
- v0.2-alpha：模板与作品基础增强完成。
- v0.2-beta：队列增强、错误标准化、导出兼容完成。
- v0.2-rc：补齐测试、修复回归、冻结范围。
- v0.2 正式版：完成发布验收清单并对外可用。

## 15. 待确认问题
1. 模板分类是固定分组还是可自定义分组。
2. 作品详情页的信息密度是否需要进一步分层。
3. `metadata.json` 是否需要扩展更多工程字段。
4. 后续测试框架与测试组织方式如何确定。
5. 后端代理能力应在 v0.2 还是 v0.3 正式落地。

## 16. 最终结论
v0.2 的核心不是增加一个新入口，而是把现有生图能力整理成稳定、可复用、可管理的创作工作台，并为后续产品化预留清晰边界。
